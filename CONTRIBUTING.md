# Contributing

Thanks for taking time to improve a Paravane Labs project.

This file provides default guidance for public repositories. Repository-specific contribution instructions take precedence when present.

## Before opening an issue

Please check whether the repository already has:

- Existing issues or discussions covering the same topic.
- Project-specific setup instructions.
- A security policy for sensitive reports.

Do not include secrets, API keys, tokens, customer data, private logs, or sensitive infrastructure details in public issues.

## Good issues include

- A clear summary of the problem or request.
- Steps to reproduce, when applicable.
- Expected behavior and actual behavior.
- Relevant versions, operating system, browser, runtime, or API endpoint details.
- Minimal logs or screenshots with sensitive data removed.

## Pull requests

Good pull requests are focused, explain the reason for the change, and include tests or verification notes when appropriate.

Before opening a pull request:

- Keep the change scoped to one concern.
- Follow the style already used by the repository.
- Update documentation when behavior changes.
- Include a clear summary of what changed and why.
- Mention any testing performed.

## Security reports

Do not open public issues for vulnerabilities. Use the process in `SECURITY.md`.
